# Rutabien — Project Context for Claude Code

## What this is
Rutabien ("every step, mapped") is a web app helping non-EU international students navigate moving to Barcelona — visas, document tracking, sworn translation referrals, legal referral, and post-arrival settling-in guidance. Full product spec lives in `MVP_Draft.md` and `Rutabien_Pitch_and_Brand.md` at the repo root, with visual/UI direction in `Rutabien_Design_Brief_1.md` — **read those before making product/logic decisions**, this file is a condensed reference, not the source of truth.

## Stack (reused from Mindshift)
- Next.js on Vercel — **region must be set to Frankfurt (EU)**, not left on US default
- Drizzle ORM + Neon Postgres — **region must be Frankfurt (EU)**, sign Neon's DPA
- Resend for transactional email / reminders
- File storage: not yet decided — see "Open decisions" below, do not assume a provider
- Payment: PayPal + credit/debit card (Stripe or similar) — **keep in sandbox/test mode** until told otherwise, see "Do not do yet" below

## Core architectural principle: two-tier data model
Do not conflate these two concerns:
1. **The Requirements table** — the content engine. `nationality × visa-type × student-status → document list`, each row carrying `translation_required`, `legalization_chain`, `home_country_apostille_authority`, `notarization_required`, `validity_window_days`, `official_source_link`, `funds_formula`, conditional rules (e.g. duration-dependent). This is authored content (by the founder + legal reviewer), not user data.
2. **Per-user tracking tables** — Users, Documents, Roadmap Progress, Support Escalation Log. This is what a specific user has done, generated by querying the Requirements table against their intake answers.

A user's roadmap is **generated by querying table 1 with table 2's answers** — never hardcode nationality-specific logic directly in application code. Adding a new nationality should mean adding new rows to the Requirements table, not writing new code paths.

## Intake is a decision tree, not a fixed form
Every question branches the next question — see `MVP_Draft.md` section 4 for the exact 9-question tree (EU/EEA gate → new/returning gate → nationality → acceptance letter → family members → part-time work → housing → arrival date → email). Each gate should skip irrelevant downstream questions entirely, not just tag data for later filtering. The progress indicator must reflect each user's actual path length, not a fixed total.

## Auth: hardened passwordless, not a simple magic link
No password field anywhere. But also **not a static, reusable, non-expiring link** — that's a real security gap for a product holding passport scans and financial documents. Required pattern:
- User requests access via email on an "Access your roadmap" page
- Link expires in ~15–30 minutes, single use
- After login, a normal secure session takes over
- **Step-up verification** (fresh one-time code) required specifically before viewing/downloading any uploaded document, even within an active session
- Documents encrypted at rest

## Critical product principles — do not violate these
- **No AI-generated roadmap content.** All roadmap/requirement content is human-authored and legal-reviewed. AI may be used for narrow utility functions (e.g. OCR to detect which document was uploaded) but never to generate or paraphrase requirement guidance shown to users.
- **Indemnification language is standing, not a one-time disclaimer.** Every roadmap screen showing requirement content needs a visible note that guidance is based on official published sources, not a guarantee of approval. Every specific number cited (fund thresholds, processing times) is a candidate to go stale — flag these for review, don't treat them as permanent constants.
- **Live support has a hard operational boundary.** Support chat can help with navigation/platform questions. It must NOT assess document sufficiency or predict outcomes — that routes to the legal-partner escalation path. This boundary should be enforced by a defined escalation trigger in code/config, not left to agent discretion.
- **The legal-partner screen leads with the role, not the individual.** Headline: "Rutabien's Legal Partner." Name/photo/credential as supporting detail underneath. This is deliberate — do not swap the hierarchy.
- **Pricing must always show a full total before checkout.** No price that changes after the user commits, no fees revealed at a later step. Applies to every checkout in the app (platform tier, translation, future lawyer consult).
- **Document Vault stays folder-organized, individual files — never merge into one PDF.** Users need to send/print single documents independently.

## Business logic specifics
- **Two pricing tiers, one-time payment, not subscriptions:** Essential (€39) — roadmap, tracking, reminders. Complete (€89, valid 12 months, not auto-renewing) — adds Document Vault, Bienvenido, extended live support, priority translation referral.
- **Translation flow has two entry points:** (1) triggered from a document flagged "Translation: Required" inside a user's roadmap, (2) a fully standalone path from the landing page ("Just need a translation?") that skips the entire intake questionnaire — translation doesn't need nationality/visa-type logic at all.
- **Nigeria is the only populated Requirements-table nationality right now.** Every other nationality should get honest general guidance, not silently wrong Nigeria-specific content.

## Do not do yet (blocked pending founder + legal partner sign-off)
- Do not enable real payment processing outside sandbox/test mode
- Do not publish any Requirements-table content as "final" without an explicit sign-off flag being set
- Do not implement a percentage-based housing referral fee — this may conflict with Spanish tenant-protection law and is unresolved
- Do not hardcode a named translator or law firm partner's real business terms — these are still being finalized
